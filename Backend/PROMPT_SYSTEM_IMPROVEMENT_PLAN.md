# RABA Prompt System Improvement Plan

## Executive Summary

This document outlines critical improvements needed for RABA's prompt generation system. Currently, tool-specific prompt templates stored in the database are **not being used** during video generation. Instead, agents rely on hardcoded vocabulary dictionaries, preventing tool customization and optimization. This plan addresses the integration gap and introduces advanced features for production-grade video generation.

## Current System Analysis

### How Prompts Are Currently Generated

#### Script Writing (`script_writer.py`)
- **Current Method**: Uses hardcoded `TOOL_VISUAL_VOCABULARY` dictionary
- **Location**: Lines 37-97 in `script_writer.py`
- **Function**: `_build_full_script_prompt()` constructs prompts dynamically using:
  - Tool category (surreal_realism, high_octane_anime, stylized_3d)
  - Hardcoded style keywords, camera styles, mood keywords
  - Research data and intent metadata
- **Issue**: Tool templates from database are completely ignored

#### Image Generation (`image_generator.py`)
- **Current Method**: Uses hardcoded `TOOL_VISUAL_VOCABULARY` dictionary
- **Location**: Lines 45-132 in `image_generator.py`
- **Function**: `build_image_prompt()` constructs prompts using:
  - Tool category to select vocabulary
  - Scene descriptions from script
  - Hardcoded style, mood, and lighting keywords
- **Issue**: `image_prompt_template` from selected tool is never used

#### Video Generation (`video_generator.py`)
- **Current Method**: Uses hardcoded `TOOL_VIDEO_VOCABULARY` dictionary
- **Location**: Lines 47-105 in `video_generator.py`
- **Function**: `build_video_prompt()` constructs prompts using:
  - Tool category to select vocabulary
  - Script output and segment information
  - Hardcoded camera movements, audio cues, lighting
- **Issue**: `video_prompt_template` from selected tool is never used

### Tool Selection Flow

1. **IntentToolSelectorAgent** selects a tool based on user topic
2. **Tool metadata** (including prompt templates) is stored in workflow state
3. **Agents receive** `selected_tool` dictionary with `category` field
4. **Agents use only** `category` to select hardcoded vocabulary
5. **Templates** (`script_prompt_template`, `image_prompt_template`, `video_prompt_template`) are **ignored**

### Tool Template Storage

- **Location**: `tools` table in Supabase database
- **Fields**: 
  - `script_prompt_template` (text)
  - `image_prompt_template` (text)
  - `video_prompt_template` (text)
- **Creation**: Generated by `ToolEnhancer` service when tools are created/updated
- **Usage**: Currently only used by `/tools/execute` API endpoint (not in main workflow)

## Critical Issues Identified

### Issue 1: Tool Templates Are Not Integrated
**Severity**: CRITICAL

- Tool-specific prompt templates exist in database but are never used
- All tools generate identical prompts (only category differs)
- No way to customize prompts per tool without code changes
- Tool improvements via `ToolEnhancer` don't affect actual generation

**Impact**:
- Cannot optimize prompts for specific tools
- Cannot A/B test different prompt strategies
- Tool customization is meaningless
- Database templates are wasted storage

### Issue 2: Hardcoded Vocabulary Dictionaries
**Severity**: HIGH

- Three hardcoded dictionaries in agent files:
  - `TOOL_VISUAL_VOCABULARY` (script_writer.py, image_generator.py)
  - `TOOL_VIDEO_VOCABULARY` (video_generator.py)
- Changes require code deployment
- No version control for prompt changes
- Difficult to maintain consistency

**Impact**:
- Slow iteration cycle for prompt improvements
- Risk of breaking changes
- No audit trail for prompt modifications

### Issue 3: No System-Driven Prompt Optimization
**Severity**: MEDIUM

- Current system only supports user feedback-based improvements
- No mechanism for bulk prompt updates
- No way to apply system-wide optimizations
- Cannot update prompts based on performance metrics

**Impact**:
- Cannot leverage system learnings
- Manual, one-by-one updates required
- No automated optimization pipeline

### Issue 4: Prompt Template Quality Issues
**Severity**: MEDIUM

Based on analysis of existing tool templates, several quality issues:

- **Inconsistent Structure**: Templates vary in detail level
- **Missing Technical Specs**: Some lack camera movements, lighting details
- **Placeholder Gaps**: Not all templates properly use available context
- **Style Drift Risk**: No mechanism to maintain consistency across scenes

## Required System Behavior

### Core Requirement: Tool Template Integration

**When a tool is selected:**
1. System retrieves tool from database (including all three prompt templates)
2. Tool templates are stored in workflow state
3. **Script Writer Agent** uses `script_prompt_template` (not hardcoded vocabulary)
4. **Image Generator Agent** uses `image_prompt_template` (not hardcoded vocabulary)
5. **Video Generator Agent** uses `video_prompt_template` (not hardcoded vocabulary)
6. Templates are rendered with context (topic, tone, duration, scene descriptions, etc.)
7. If template is missing, fall back to hardcoded vocabulary (backward compatibility)

### Template Rendering Context

Each template should have access to:

**Script Template Context:**
- `{topic}` - Video topic
- `{tone}` - Desired tone (serious, humorous, dramatic, casual)
- `{duration}` or `{duration_seconds}` - Video duration
- `{intent_type}` - Intent classification (educational, entertainment, etc.)
- `{target_audience}` - Target audience
- `{research_summary}` - Research findings summary
- `{tool_category}` - Tool category name
- `{tool_name}` - Tool display name

**Image Template Context:**
- `{scene_description}` - Scene visual description from script
- `{style}` - Tool category/style
- `{scene_number}` - Current scene number
- `{total_scenes}` - Total number of scenes
- `{camera_direction}` - Camera direction from script
- `{lighting}` - Lighting description from script
- `{mood}` - Mood from script
- `{topic}` - Video topic
- `{duration_seconds}` - Video duration

**Video Template Context:**
- `{script}` - Full script output (formatted)
- `{duration}` or `{duration_seconds}` - Video duration
- `{tool_category}` - Tool category
- `{tool_name}` - Tool display name
- `{segment_info}` - Segment timing information (for multi-segment videos)
- `{previous_segment_end}` - Description of previous segment end (for extensions)

### Fallback Strategy

- If tool template is `NULL` or empty, use hardcoded vocabulary
- Log warning when fallback is used
- Track which tools use fallbacks for prioritization

### User-Controlled Features

**Critical Requirement**: Subtitles and audio must be strictly controlled by user input.

**Subtitles Behavior**:
- Subtitles are **ONLY** generated if user explicitly enables them in workflow input
- Default behavior: Subtitles are **DISABLED** (no subtitles generated)
- When disabled: Video prompts must explicitly exclude subtitles ("no subtitles, no text overlays")
- When enabled: OverlayGeneratorAgent generates subtitle overlays programmatically
- User input field: `enable_subtitles` (boolean, default: false)

**Audio Behavior**:
- Audio is **ONLY** generated if user explicitly enables it in workflow input
- Default behavior: Audio is **DISABLED** (no audio generated)
- When disabled: Video prompts must explicitly request silent video ("no audio, no sound, silent video")
- When enabled: Veo generates audio according to script and video template
- User input field: `enable_audio` (boolean, default: false)

**Implementation Requirements**:
- Both `enable_subtitles` and `enable_audio` must be part of WorkflowInput model
- Both flags must be passed through workflow state
- Video Generator must check these flags before generating video
- Video prompts must be modified based on these flags
- OverlayGeneratorAgent only runs if `enable_subtitles` is true
- Veo service must respect `enable_audio` flag in generation config
- All agents must log the state of these flags for monitoring

**State Management**:
- Store `enable_subtitles` and `enable_audio` in workflow state
- Pass flags through all state transitions
- Include flags in logging for debugging

## Improvement Plan

## Phase 1: Tool Template Integration

### 1.1 Create Prompt Builder Service

**Purpose**: Centralized service for rendering tool templates with context

**Components Needed**:
- Template renderer with placeholder substitution
- Context builder that aggregates all available data
- Template validator (checks for required placeholders)
- Fallback handler (uses hardcoded vocabularies when template missing)

**Responsibilities**:
- Accept tool template string and context dictionary
- Replace placeholders with actual values
- Handle missing placeholders gracefully
- Merge template with additional context (scene data, script data)
- Return fully rendered prompt
- **Log all operations**: Template rendering start/end, placeholder substitutions, missing placeholders, fallback triggers

**Integration Points**:
- Called by Script Writer Agent before prompt construction
- Called by Image Generator Agent before prompt construction
- Called by Video Generator Agent before prompt construction

### 1.2 Modify Script Writer Agent

**Changes Required**:
- Remove dependency on hardcoded `TOOL_VISUAL_VOCABULARY`
- Retrieve `script_prompt_template` from `selected_tool` in state
- Use Prompt Builder Service to render template
- Build context dictionary with: topic, tone, duration, intent_type, target_audience, research_summary, tool_category, tool_name
- Pass rendered template to Gemini (instead of dynamically built prompt)
- Fallback to current hardcoded method if template missing
- **Add comprehensive logging**: Log template retrieval, rendering, context building, and fallback usage

**Preserve**:
- System instruction building (`_build_system_instruction`)
- Research summary extraction
- Viral metrics calculation
- All existing functionality except prompt construction

### 1.3 Modify Image Generator Agent

**Changes Required**:
- Remove dependency on hardcoded `TOOL_VISUAL_VOCABULARY`
- Retrieve `image_prompt_template` from `selected_tool` in state
- For each scene, use Prompt Builder Service to render template
- Build context dictionary with: scene_description, style, scene_number, total_scenes, camera_direction, lighting, mood, topic, duration_seconds
- Pass rendered template to Nano Banana service
- Fallback to current hardcoded method if template missing
- **Add comprehensive logging**: Log template retrieval per scene, rendering, image generation progress, and timing

**Preserve**:
- Image count calculation
- Style reference building
- Sequential generation logic
- Supabase storage upload
- All existing functionality except prompt construction

### 1.4 Modify Video Generator Agent

**Changes Required**:
- Remove dependency on hardcoded `TOOL_VIDEO_VOCABULARY`
- Retrieve `video_prompt_template` from `selected_tool` in state
- For each segment, use Prompt Builder Service to render template
- Build context dictionary with: script (formatted), duration, tool_category, tool_name, segment_info, previous_segment_end (for extensions)
- Check `enable_audio` flag from workflow state and modify prompt accordingly
- Check `enable_subtitles` flag from workflow state and modify prompt accordingly
- Pass rendered template to Veo service
- Fallback to current hardcoded method if template missing
- **Add comprehensive logging**: Log template retrieval, audio/subtitle flags, segment generation progress, and timing

**Preserve**:
- Segment planning logic
- Reference image selection
- Multi-segment generation
- Video extension logic
- All existing functionality except prompt construction

### 1.5 Update Workflow State

**Changes Required**:
- Ensure `selected_tool` dictionary includes all three template fields
- Verify templates are passed through workflow state transitions
- Add validation to ensure templates are available when needed

### 1.6 Testing Requirements

**Test Cases**:
- Tool with all three templates → All agents use templates
- Tool with missing script template → Script Writer uses fallback
- Tool with missing image template → Image Generator uses fallback
- Tool with missing video template → Video Generator uses fallback
- Tool with NULL templates → All agents use fallbacks
- Template with missing placeholders → Graceful handling
- Template with all placeholders → Successful rendering
- Verify backward compatibility with existing workflows

## Phase 2: Prompt Management System

### 2.1 Bulk Prompt Update Endpoint

**Endpoint**: `POST /api/tools/prompts/bulk-update`

**Purpose**: Update prompt templates for multiple tools at once based on system optimizations

**Request Schema**:
- `tool_ids`: Array of tool IDs to update
- `update_type`: Enum ("all", "script_only", "image_only", "video_only", "script_and_image", "script_and_video", "image_and_video")
- `improvement_reason`: String describing why prompts are being improved (e.g., "Performance optimization based on success_rate analysis")
- `prompts`: Object with optional `script_prompt_template`, `image_prompt_template`, `video_prompt_template`
- `use_ai_enhancement`: Boolean - if true, use Gemini to enhance prompts based on tool metadata
- `preserve_structure`: Boolean - if true, maintain existing template structure while improving content

**Response Schema**:
- `updated_count`: Number of tools successfully updated
- `failed_updates`: Array of tool IDs that failed to update with error messages
- `updated_tools`: Array of updated tool metadata
- `improvement_summary`: Summary of changes made

**Behavior**:
- If `use_ai_enhancement` is true, call ToolEnhancer service for each tool to generate improved templates
- If `use_ai_enhancement` is false, use provided prompts directly
- Update database records
- Increment tool version number
- Add entry to improvement_history
- Update `last_improved_at` timestamp
- Return success/failure for each tool
- **Log all operations**: Request received, tool processing start/end, AI enhancement calls, database updates, success/failure for each tool

**Validation**:
- Verify all tool IDs exist
- Validate prompt template format (check for required placeholders)
- Ensure templates meet minimum quality standards (length, detail level)
- Check for SQL injection risks in prompts

### 2.2 Individual Tool Prompt Update Endpoint

**Endpoint**: `PUT /api/tools/{tool_id}/prompts`

**Purpose**: Update prompt templates for a single tool based on system optimizations

**Request Schema**:
- `update_type`: Enum (same as bulk update)
- `improvement_reason`: String
- `prompts`: Object with optional template fields
- `use_ai_enhancement`: Boolean
- `preserve_structure`: Boolean

**Response Schema**:
- `tool_id`: Updated tool ID
- `updated_templates`: Object showing which templates were updated
- `improvement_summary`: Summary of changes
- `new_version`: New tool version number

**Behavior**:
- Same as bulk update but for single tool
- More detailed response with before/after comparison
- **Log all operations**: Request received, tool processing, before/after template comparison, validation results

### 2.3 System-Driven Improvement Service

**Purpose**: Automated service that analyzes tool performance and suggests prompt improvements

**Components**:
- Performance analyzer (reads success_rate, usage_count, viral_score metrics)
- Improvement generator (uses Gemini to suggest better prompts)
- Quality validator (ensures improved prompts meet standards)
- Update orchestrator (executes bulk updates)

**Analysis Criteria**:
- Tools with low success_rate (< 0.5) → High priority for improvement
- Tools with high usage_count but low success_rate → Optimization opportunity
- Tools with declining success_rate over time → Regression detection
- Tools with missing templates → Template generation needed

**Improvement Process**:
1. Analyze tool performance metrics
2. Identify tools needing improvement
3. Generate improved prompts using Gemini (with tool context)
4. Validate improved prompts
5. Present recommendations to system admin
6. Execute bulk update when approved

**Integration Points**:
- Can be triggered manually via admin endpoint
- Can be scheduled (cron job) for periodic analysis
- Can be triggered by monitoring service when metrics drop

### 2.4 Prompt Template Quality Standards

**Minimum Requirements for All Templates**:

**Script Template**:
- Must include: {topic}, {tone}, {duration}
- Minimum 150 words
- Must specify hook requirements (first 2-3 seconds)
- Must include pattern interrupt guidance
- Must specify word count guidance
- Must include viral signal patterns

**Image Template**:
- Must include: {scene_description}, {style}
- Minimum 150 words
- Must specify artistic style direction
- Must include lighting requirements
- Must include color palette definition
- Must include composition/framing guidance
- Must include technical specs (resolution, quality)

**Video Template**:
- Must include: {script}, {duration}
- Minimum 150 words
- Must specify camera movements
- Must include camera angle guidance
- Must specify pacing/temporal elements
- Must include visual effects guidance
- Must include audio direction
- Must specify consistency requirements

**Validation Rules**:
- Check placeholder presence
- Verify minimum word count
- Ensure technical specifications are included
- Validate structure completeness

## Phase 3: Advanced Video Generation Architecture

Based on production-grade recommendations, implement the following improvements:

### 3.1 Global Style Anchor Agent

**Purpose**: Prevent style drift across scenes by establishing a master style reference

**Implementation**:
- New agent: `StyleArchitectAgent`
- Runs after tool selection, before image generation
- Uses Nano Banana Pro to generate a Master Style Sheet image
- Master Style Sheet contains:
  - Color palette (HEX codes extracted and stored)
  - Lighting profile (text description)
  - Texture reference (text description)
  - Visual style guide (text description)
- **Add comprehensive logging**: Log agent start/end, style sheet generation progress, color palette extraction, style guide generation

**Integration**:
- Master Style Sheet image passed as `reference_image` to all subsequent image generations
- Style guide text prepended to all image and video prompts
- Stored in workflow state as `master_style_sheet`

**Workflow Position**:
- After: Tool Selection
- Before: Script Writing (style guide influences script visual directions)
- Used by: Image Generator, Video Generator

**Output Schema**:
- `style_sheet_image_url`: URL of generated style sheet
- `color_palette`: Array of HEX codes
- `lighting_profile`: Text description
- `texture_reference`: Text description
- `style_guide_text`: Full style guide text for prompts

### 3.2 Decouple Text and UI from Video

**Purpose**: Generate clean video plates without text, then composite text overlays programmatically

**Implementation**:
- Modify Video Generator to explicitly exclude text/UI in prompts
- Add negative prompt instructions: "no text, no subtitles, no UI elements, no labels"
- New agent: `OverlayGeneratorAgent`
- New service: `VideoCompositorService`

**OverlayGeneratorAgent**:
- **Only runs if `enable_subtitles` is true** (check workflow state)
- Generates transparent overlay images for text/subtitles
- Uses Nano Banana Pro or standard graphics library
- Creates overlays based on script dialogue and visual cues
- Outputs: Array of overlay images with timing information
- Logs when subtitles are generated or skipped

**VideoCompositorService**:
- **Only runs if `enable_subtitles` is true**
- Uses FFmpeg or cloud video editor API
- Composites overlay images onto video plates
- Handles timing synchronization
- Ensures 100% spelling accuracy (text is programmatic, not AI-generated)
- If subtitles disabled, returns clean video unchanged
- Logs compositing operations

**Workflow Position**:
- After: Video Generation
- Before: Final Output

**Changes to Video Generator**:
- Check `enable_subtitles` flag from workflow state
- If `enable_subtitles` is false: Add to video prompts "Generate clean video without any text, subtitles, UI elements, or labels"
- If `enable_subtitles` is true: Generate clean video, then pass to OverlayGeneratorAgent
- Store "clean_video_url" in state
- Pass to compositor service only if subtitles are enabled
- Log subtitle generation decision

### 3.3 Character Seed Protocol

**Purpose**: Maintain character consistency across scenes when recurring characters appear

**Implementation**:
- Enhance Script Writer to identify "Lead Character" in script output
- New agent: `CharacterReferenceGeneratorAgent`
- New field in ScriptOutput: `lead_character` (optional)
- **Add comprehensive logging**: Log character identification, reference generation progress, image generation for each view (front, side, back, face)

**CharacterReferenceGeneratorAgent**:
- Runs after script generation if lead_character is identified
- Uses Nano Banana Pro to generate Character Reference Sheet
- Creates multiple views: front, side, back, close-up face
- Stores reference sheet images

**Integration**:
- Character reference images passed to all image generations
- Character reference images passed to all video generations
- Used as `character_ref` or `reference_image` in Veo prompts
- Stored in workflow state as `character_reference_sheet`

**Workflow Position**:
- After: Script Writing (if lead_character identified)
- Before: Image Generation
- Used by: Image Generator, Video Generator

**Output Schema**:
- `character_name`: Name of lead character
- `character_description`: Text description
- `reference_images`: Array of reference image URLs (front, side, back, face)
- `character_metadata`: Additional character details

### 3.4 Physics & Scale Validator

**Purpose**: Flag visual logic issues in scripts before generation

**Implementation**:
- New agent: `VisualLogicValidatorAgent`
- Uses Gemini 2.5 Pro to analyze script for visual logic issues
- Identifies "High Hallucination Risk" scenarios:
  - Precise numerical representations (e.g., "400-mile tape measure")
  - Complex spatial relationships
  - Unrealistic scale representations
  - Physically impossible scenarios

**Validation Process**:
1. Analyze script scenes for visual logic issues
2. Flag high-risk scenarios
3. Generate alternative visual suggestions
4. If critical issues found, request script revision
5. Pass validated script to image/video generation
- **Add comprehensive logging**: Log validation start/end, flagged issues count and details, risk level assessment, script revision requirements

**Integration**:
- Runs after Script Writing
- Before Image Generation
- Can trigger Script Writer revision if critical issues found

**Output Schema**:
- `validation_passed`: Boolean
- `flagged_issues`: Array of issue descriptions
- `suggested_alternatives`: Array of alternative visual descriptions
- `risk_level`: Enum ("low", "medium", "high", "critical")
- `requires_revision`: Boolean

### 3.5 Temporal Buffer Generation

**Purpose**: Generate extra footage to remove jitter at clip boundaries

**Implementation**:
- Modify Video Generator to request 20% more footage per scene
- New agent: `TrimAgent`
- New service: `VideoTrimmerService`

**Video Generator Changes**:
- Calculate buffer: `requested_duration * 1.2`
- Request extended duration from Veo
- Store extended video in state

**TrimAgent**:
- Analyzes video segments
- Identifies start/end jitter zones (first/last 0.5 seconds)
- Generates trim instructions

**VideoTrimmerService**:
- Uses FFmpeg to trim videos
- Removes first 0.5s and last 0.5s from each segment
- Ensures smooth transitions
- Outputs final trimmed video
- **Add comprehensive logging**: Log trim analysis, jitter zone identification, trim operations per segment, final output

**Workflow Position**:
- After: Video Generation
- Before: Final Output

**Configuration**:
- Buffer percentage: Configurable (default 20%)
- Trim duration: Configurable (default 0.5s per end)

## Phase 4: Prompt Template Optimization

### 4.1 Enhanced Template Structure

Based on analysis of existing templates, improve template quality standards:

**Script Template Enhancements**:
- Add explicit timing breakdowns (e.g., "[HOOK: 0-3s]", "[NARRATIVE: 3-15s]")
- Include word count guidance (e.g., "Keep voiceover between 45-55 words")
- Specify pattern interrupt frequency (e.g., "pattern interrupt every 4 seconds")
- Add emotional arc requirements
- Include viral signal patterns

**Image Template Enhancements**:
- Require specific art direction references (e.g., "MAPPA and Ufotable style")
- Mandate lighting specifications (e.g., "Volumetric god-rays, rim lighting in neon gold")
- Require color palette with specific colors (e.g., "Royal purples, deep golds, electric magentas")
- Include composition details (e.g., "Extreme low-angle 'worm's eye' perspective")
- Add technical specifications (e.g., "8K resolution, sharp focus on facial micro-expressions")

**Video Template Enhancements**:
- Specify camera movement sequences (e.g., "bullet-time pans, rapid 360-degree arc shots")
- Include pacing instructions (e.g., "Start with slow-motion tension, ramp up to high-frame-rate action")
- Require audio direction (e.g., "Orchestral dubstep hybrid with heavy bass drops")
- Add visual effects specifications (e.g., "calligraphic ink strokes during movement peaks")
- Include consistency requirements (e.g., "Match exact visual style, characters, and atmosphere")

### 4.2 Template Validation Service

**Purpose**: Ensure all templates meet quality standards before use

**Validation Rules**:
- Check for required placeholders
- Verify minimum word count (150 words)
- Ensure technical specifications are present
- Validate structure completeness
- Check for style drift prevention instructions
- Verify consistency requirements

**Integration**:
- Run during tool creation/update
- Run during bulk prompt updates
- Run before template usage (runtime validation)
- **Add comprehensive logging**: Log validation start/end, rules checked, validation failures with details, quality scores, approval/rejection decisions

### 4.3 Template Improvement Guidelines

**For ToolEnhancer Service**:
- Update system prompts to enforce new template standards
- Include examples of high-quality templates
- Emphasize specificity and technical details
- Require style consistency instructions

**For Bulk Update Service**:
- When using AI enhancement, ensure improved templates meet new standards
- Compare old vs new templates for quality improvement
- Reject improvements that don't meet minimum standards
