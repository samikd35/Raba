# Module 3: MVP Requirement Generator — Templates Requirements

## Prerequisites

By this stage, the following have already been completed:

- ✅ Validated the problem through the **Problem Validation Report**
- ✅ Defined up to two **Personas** and their **Customer Profiles** (Pains, Gains, Jobs)
- ✅ Crafted **Hypotheses & Assumptions**, ran **Market Research**, and refined the Customer Profiles
- ✅ Completed **Value Map** (Pain Relievers, Gain Creators, Products & Services)
- ✅ Built **VPS v1**, **BMC v1**, received a **Solution Critique**, and refined to **VPS v2** & **BMC v2**

Now, in Module 3, all that work is translated into clear, testable MVP Product Requirements.

## Template Guide Overview

For each core product type below, the following prompts guide the description of:

| Aspect | Description |
|--------|-------------|
| **Core Flows** | What the MVP must do |
| **Postponed Items** | What it must not yet do |
| **Assumptions** | What assumptions it is designed to test |
| **Success Metrics** | What metrics will indicate if the MVP is working |

## A1. Software / SaaS Product (Web or Mobile App)

**Examples:** CRM for clinics, farmer advisory app, school management system.

When defining product requirements for an MVP SaaS/app, specify clearly:

### MVP Purpose and Scope

#### Purpose
Briefly describe why this MVP is being built now, and which validated problem & Persona (from VPC v2 and VPS v2) it is focused on.

#### Objective
What are the main learning goals or outcomes the Entrepreneur(s) want from this MVP (e.g., validate willingness to pay, validate daily usage, validate a specific workflow)?

#### Scope
Clearly state what is in scope for this MVP (user segments, core flows, geography) and what is out of scope for now.

### Primary User Persona

Summarize the primary Persona that this MVP serves first and describe the single most important job this product helps that Persona accomplish at this stage.

### Product Key Features

Using the refined Customer Profile & Value Map (VPC v2), Value Proposition Statement v2 (VPS v2) and Business Model Canvas (BMC), list the key features of the solution.

Split them into:

- **Must-haves for MVP** — The minimum features needed to deliver the promised value and test the riskiest assumptions.
- **Nice-to-haves for later** — Improvements or extra features that support the value but are not required for initial learning.

For each feature (both must-have and nice-to-have):
- Specify which **Job to be Done** it supports
- Categorize that job as **Functional**, **Emotional**, or **Social** (e.g., "Feature X → Functional job: save time on reporting")

> Make sure each must-have feature connects clearly to a Job, Pain or Gain in the VPC and to a relevant revenue/cost or key activity block in the BMC.

### Critical User Flows (MVP Only)

Describe the minimum end-to-end flow that a user must complete to receive value from the product (for example, sign up → add specific data → receive a defined outcome), and clarify which flows are considered must-have for the initial launch versus nice-to-have enhancements for later iterations.

### Data & Information Needed

Specify the essential data that users must provide (such as key fields, forms, or uploads) and the information the system must store, process, and present back (dashboards, alerts, reports) in order to deliver the intended value.

### Automation & Intelligence Level

Explain which parts of the workflow will remain manual in the MVP and which parts will be automated, including any planned use of AI or algorithms and the fallbacks or manual processes that will be used when automation is not available or fails.

### Access & Platform Choices

Clarify whether the MVP will be web-only, mobile-only, or available on both, provide the rationale for this choice, and define how users will access the product (for example, via login, simple link, or access code).

### Constraints & Non-Goals

List the complex features, integrations, and advanced capabilities that will deliberately not be included in the MVP and state any compliance or privacy constraints that must already be respected, especially for sensitive domains such as health or education.

### Success Signals

Define two to three quantitative indicators that will be used to judge whether the MVP is working for this SaaS/product (for example, number of active users, frequency of completed key workflows, or short-term retention) and describe the types of qualitative feedback from users that would indicate the product is meaningfully improving their work or lives.

## A2. Digital Content / Course / EdTech Product

**Examples:** Online academy, cohort-based course, video lesson platform, test prep app.

For a learning or content-focused MVP, define:

### MVP Purpose and Scope

#### Purpose
Why this learning product exists at MVP stage and which specific learner Persona and problem (from VPC v2 / VPS v2) it is addressing first.

#### Objective
The concrete learning or behaviour change that should be observed as a result of this MVP.

#### Scope
The topics, formats, and learner segments included in this MVP, and those explicitly postponed.

### Learner Persona and Outcomes

Describe the initial learner Persona targeted by the MVP and articulate the specific learning outcomes, behavioural changes, or practical capabilities that learners should achieve after engaging with the content.

### Product Key Features

Using the refined Customer Profile & Value Map (VPC v2), Value Proposition Statement v2 (VPS v2) and Business Model Canvas (BMC), list the key features of the learning product.

Split them into:

- **Must-haves for MVP** — The minimum content, interactions, and tools needed to deliver the promised learning outcomes and test the riskiest assumptions.
- **Nice-to-haves for later** — Additional formats, modules, or tools that enhance engagement but are not required for initial learning.

For each feature (both must-have and nice-to-have):
- Specify which **Job to be Done** it supports
- Categorize that job as **Functional**, **Emotional**, or **Social**

> Make sure each must-have feature connects clearly to a Job, Pain or Gain in the VPC and to a relevant revenue/cost or key activity block in the BMC.

### Core Learning Journey (End-to-End Workflow)

Outline the minimum learning journey for the MVP, including the number and structure of modules, lessons, and assessments, and indicate whether the experience will be self-paced, cohort-based, or a hybrid model.

### Content Format and Delivery

Specify the essential content formats to be included in the MVP (such as video, text, quizzes, or live sessions) and the primary platforms or devices through which learners will access the material.

### Engagement and Support

Describe how learners will seek help or clarification during the learning journey and what feedback or support mechanisms will be available at MVP stage, such as chat, forums, office hours, or assignment reviews.

### Assessment and Proof of Learning

Explain how understanding and application of the material will be assessed and whether any formal proof of learning—such as certificates, badges, or other recognitions—will be provided in the MVP or added later.

### Operational Requirements

Summarize who will be responsible for creating, reviewing, and maintaining content during the MVP and note any institutional or commercial partners involved in delivery, such as schools or companies.

### Success Signals

Identify the key metrics that will indicate strong learning and engagement (for example, module completion rates, assessment performance, or session attendance) and describe the types of learner feedback or stories that will demonstrate that the product is achieving the intended outcomes.

## A3. Platforms and Marketplaces (Two-Sided and Multi-Sided)

**Examples:** Plumber marketplace, farmer marketplace, landlord-tenant platform, job platforms, mobility platforms, and other multi-party coordination platforms.

For any platform or marketplace MVP (two- or multi-sided), define:

### MVP Purpose and Scope

#### Purpose
The core matching or coordination problem being solved, for which Personas on each side, based on VPC v2 and VPS v2.

#### Objective
The learning goals regarding liquidity, matching quality, and willingness to transact on the platform.

#### Scope
The geography, vertical, and user types included in this first version, and those intentionally excluded.

### Platform Focus and Niche

Define the different sides of the platform, such as buyers and sellers or employers and job seekers, state which side will be prioritized in the MVP, and describe the narrow initial niche in terms of segment, geography, and use case that will be targeted to enable realistic matching.

### Product Key Features

Using the refined Customer Profile & Value Map (VPC v2), Value Proposition Statement v2 (VPS v2) and Business Model Canvas (BMC), list the key features of the platform.

Split them into:

- **Must-haves for MVP** — The minimum features needed to enable discovery, matching, and basic trust between sides while testing key assumptions about the marketplace model.
- **Nice-to-haves for later** — Richer discovery, reputation, or automation features that can be layered on once basic liquidity is proven.

For each feature (both must-have and nice-to-have):
- Specify which **Job to be Done** it supports
- Categorize that job as **Functional**, **Emotional**, or **Social**

> Make sure each must-have feature connects clearly to a Job, Pain or Gain in the VPC and to a relevant revenue/cost or key activity block in the BMC.

### Matching Value Flow

Describe the core interaction that creates value across the platform sides, outline the simplest possible path from a user request or search to a completed match or transaction, and clarify which steps will be handled manually versus automated in the product during the MVP.

### Onboarding Structure

Summarize the minimum information required from each user type to join the platform, the simplest onboarding process designed for low friction, and any complex roles or permissions that will be deferred and handled manually in the MVP.

### Platform Data and Discovery

Explain what data will be aggregated or centralised at MVP, the basic search, filter, or recommendation capabilities that will be available, and the essential information that must appear on listings or profiles to enable confident decisions by users.

### Trust and Policy Framework

Describe the trust mechanisms that will be present in the MVP—such as ratings, reviews, or verification—along with any eligibility rules or content guidelines that must be enforced and the lightweight processes that will be used to manage disputes, cancellations, or fraud risk.

### Payments and Transactions

Clarify how transactions will be handled in the MVP, including whether payments occur online, offline, or through a hybrid model, how transaction status will be recorded for learning and tracking, and whether the platform will hold funds or only facilitate connections between parties.

### Marketplace Launch Strategy

Outline the strategy for seeding the platform so that early users on each side experience value, including any manual interventions or operational hacks (such as curated matching or direct outreach), and describe how imbalances between supply and demand will be monitored and addressed.

### Success Signals

Specify the early metrics that will indicate platform traction—such as matches per week, conversion from request to completed job, or repeat usage—and describe how engagement on each side will be tracked and how structured feedback from users will inform improvements to liquidity and experience.

## A4. Tech-Enabled Service (Software + Operations)

**Examples:** Dossie-style digitizing records, telemedicine with nurses + platform, managed marketplace with fulfilment.

Here software is combined with heavy operations. Define:

### MVP Purpose and Scope

#### Purpose
The end-to-end outcome being delivered by combining software and operations, and the primary Persona it serves.

#### Objective
The learning goals related to feasibility, unit economics, and user experience.

#### Scope
The parts of the journey covered in this MVP (from request to outcome) and the complex cases or segments excluded for now.

### Core Service + Software Combination

Describe the division between software experience and human operations in the service, indicating which parts of the journey will rely on technology and which will be executed manually, especially in the early MVP.

### Product Key Features

Using the refined Customer Profile & Value Map (VPC v2), Value Proposition Statement v2 (VPS v2) and Business Model Canvas (BMC), list the key features of the tech-enabled service.

Split them into:

- **Must-haves for MVP** — The minimum features and service elements needed to reliably deliver the promised outcome and test the riskiest assumptions.
- **Nice-to-haves for later** — Enhancements that improve efficiency, automation, or user experience but are not essential for early validation.

For each feature (both must-have and nice-to-have):
- Specify which **Job to be Done** it supports
- Categorize that job as **Functional**, **Emotional**, or **Social**

> Make sure each must-have feature connects clearly to a Job, Pain or Gain in the VPC and to a relevant revenue/cost or key activity block in the BMC.

### End-to-End Journey

Summarize the full journey from initial customer request through service delivery to final outcome, noting at each stage whether the interaction is digital, human, or hybrid and highlighting the critical path that must work for the MVP.

### Operational Data Flows

Explain what data will be captured, processed, and returned to customers and what internal tools or systems staff will use to manage workflows relative to what customers see on the front end.

### Capacity and Service Levels

Define the throughput the operation can handle at MVP stage—such as number of records processed or consultations delivered per time period—and state the standard turnaround times and service commitments that will be honoured.

### Quality Management and Feedback

Describe the mechanisms for checking the quality of operational work and the processes through which customer feedback will be collected and used to refine both the software and service processes over time.

### Success Signals

Identify the combination of metrics—such as turnaround time, error rates, customer satisfaction scores, and repeat orders—that will be used to evaluate whether the tech-enabled service is delivering value effectively.

## A5. Fintech / Financial Product

**Examples:** Lending product for SMEs, savings product, insurance product, agent banking platform.

For a financial product MVP, it is critical to be explicit about risk, regulation, and trust. Define:

### MVP Purpose and Scope

#### Purpose
Why this financial MVP is being launched now and which specific financial pain or gap (from VPC v2 / VPS v2) it addresses first.

#### Objective
The key assumptions about risk, demand, or behaviour to be validated (such as repayment behaviour, savings consistency, or claim ratios).

#### Scope
The segments, ticket sizes, product types (loan, savings, insurance), and geographies included in this MVP, and those excluded.

### Financial Customer Profile

Describe the initial financial customer Persona targeted by the MVP and the primary financial job they are trying to accomplish, such as borrowing, saving, insuring, or transferring funds.

### Product Key Features

Using the refined Customer Profile & Value Map (VPC v2), Value Proposition Statement v2 (VPS v2) and Business Model Canvas (BMC), list the key features of the financial product.

Split them into:

- **Must-haves for MVP** — The essential features needed to deliver the financial service safely and understandably while testing the riskiest hypotheses.
- **Nice-to-haves for later** — Additional features (such as advanced analytics, gamification, or partner integrations) that enhance value but are not required for initial validation.

For each feature (both must-have and nice-to-have):
- Specify which **Job to be Done** it supports
- Categorize that job as **Functional**, **Emotional**, or **Social**

> Make sure each must-have feature connects clearly to a Job, Pain or Gain in the VPC and to a relevant revenue/cost or key activity block in the BMC.

### Core Financial Flow

Summarize the simplest end-to-end financial flow for the MVP, from application or sign-up through eligibility assessment to disbursement or activation and eventual repayment or completion, highlighting which steps can be handled manually at the beginning.

### Financial Product Structure

Define the core terms of the financial product—including amounts, pricing, fees, tenures, and coverage conditions—and describe how defaults, claims, or disputes will be handled operationally in the MVP.

### Risk and Compliance

Outline the regulatory environment relevant to the product, including licensing or approvals required, and state the minimum KYC, AML, and data protection practices that will be implemented from the start.

### Customer Protection

Describe how the product will be communicated in a clear, transparent way, without hidden fees or confusing terms, and note any guarantees, protections, or redress mechanisms that will be offered to early users.

### Technology and Operations Split

Explain which parts of the financial flow will be handled through software and which will rely on human review or partner institutions, and describe how transactions and key events will be recorded to maintain a reliable audit trail.

### Success Signals

Identify the primary metrics—such as repayment rates, savings consistency, claims ratios, active user counts, or retention—that will be monitored to judge performance and explain how edge cases like late payments or denied claims will be used to refine product design and risk policies.

Split them into:

- **Must-haves for MVP** — The minimum service components, processes, and touchpoints needed to deliver the promised value and test the riskiest assumptions.
- **Nice-to-haves for later** — Improvements or extra components that support the value but are not required for initial learning.

For each feature (both must-have and nice-to-have):
- Specify which **Job to be Done** it supports
- Categorize that job as **Functional**, **Emotional**, or **Social**

### Success Signals

Identify the indicators of service-market fit, such as repeat clients, referrals, upsells, on-time completion rates, and satisfaction scores, and describe how structured feedback from engagements will be captured and used to improve the service.

Which variants (flavours, sizes, formulations), channels, and locations are included in this first run, and which are postponed?

### Consumer Use Case

Describe the first target consumer Persona and outline the main situations and contexts in which the product will be used, including expected frequency of use.

### Product Composition

Summarize the key ingredients or materials that define the product, explain why they have been selected, and highlight the attributes that matter most at MVP stage, such as taste, scent, texture, or shelf life.

### Product Key Features

Using the refined Customer Profile & Value Map (VPC v2), Value Proposition Statement v2 (VPS v2) and Business Model Canvas (BMC), list the key features of the solution.

Split them into:

- **Must-haves for MVP** — The minimum features needed to deliver the promised value and test the riskiest assumptions.
- **Nice-to-haves for later** — Improvements or extra features that support the value but are not required for initial learning.

For each feature (both must-have and nice-to-have):
- Specify which **Job to be Done** it supports
- Categorize that job as **Functional**, **Emotional**, or **Social** (e.g., "Feature X → Functional job: save time on reporting")

> Make sure each must-have feature connects clearly to a Job, Pain or Gain in the VPC and to a relevant revenue/cost or key activity block in the BMC.

### Packaging and Formats

## C2. Hardware / IoT Product

**Examples:** Smart irrigation system, POS device, GPS tracker, medical device.

### MVP Purpose and Scope

#### Purpose
What core sensing/control outcome are the Entrepreneur(s) trying to achieve with this device + software for the first Persona?

#### Objective
What do the Entrepreneur(s) want to learn from this MVP about technical feasibility, reliability, and user adoption in the field?

#### Scope
Which environments, use cases, and user types will the Entrepreneur(s) support in this first version, and which will the Entrepreneur(s) exclude for now?

### Product Key Features

Using the refined Customer Profile & Value Map (VPC v2), Value Proposition Statement v2 (VPS v2) and Business Model Canvas (BMC), list the key features of the solution.

Split them into:

- **Must-haves for MVP** — The minimum features needed to deliver the promised value and test the riskiest assumptions.
- **Nice-to-haves for later** — Improvements or extra features that support the value but are not required for initial learning.

For each feature (both must-have and nice-to-have):
- Specify which **Job to be Done** it supports
- Categorize that job as **Functional**, **Emotional**, or **Social** (e.g., "Feature X → Functional job: save time on reporting")

> Make sure each must-have feature connects clearly to a Job, Pain or Gain in the VPC and to a relevant revenue/cost or key activity block in the BMC.

### Core Sensing/Control Job

Describe the primary sensing, monitoring, control, or tracking job that the device must perform and specify the minimum acceptable levels of accuracy, reliability, or responsiveness for the MVP.

### Device + Software Interaction
